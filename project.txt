Global Hydrology Feed System — Full Build Specification
1. Objective

Build a standalone global hydrology data feed platform that ingests river gauge observations and modeled river flows from multiple upstream providers, normalizes them into a consistent global schema, stores both normalized and raw payloads, and exposes them through a clean REST API.

This system is infrastructure only. It is not a frontend, dashboard, or map app.

The platform must support:

real observed river gauges

modeled river reaches

latest observations

historical timeseries

thresholds

warnings

provenance

raw payload retention

global geographic querying

future provider expansion

2. Design Principles
2.1 Source separation

Never mix observed and modeled data semantically.

Required distinction:

source_type = observed

source_type = modeled

Modeled discharge must never be returned as if it were a gauge reading.

2.2 Canonical normalization

Every upstream provider may use different:

identifiers

property names

units

timestamps

quality codes

geometries

station metadata conventions

The platform must translate these into one internal schema while preserving original payloads.

2.3 Provenance preservation

Every normalized record must be traceable back to:

provider

provider object ID

raw payload

ingestion timestamp

normalization version

2.4 Idempotent ingestion

All sync jobs must be safe to rerun.
Use UPSERT patterns and deterministic keys.

2.5 Global scalability

The system must support:

tens of thousands of observed stations

millions of modeled reaches

high-frequency polling

partitioned historical storage

provider-by-provider expansion

7. Canonical Units

Store both native and canonical values.

Canonical standards:

discharge → m3/s

stage → m

water_level → m

storage → m3

reservoir_storage → m3

reservoir_elevation → m

rainfall → mm

velocity → m/s

temperature → degC

7.1 Unit conversion requirements

Implement a centralized conversion library, not provider-specific ad hoc math.

Examples:

ft3/s → m3/s

kcfs → m3/s

ft → m

cm → m

ML or acre-ft → m3

If conversion is ambiguous, store:

native value

native unit

canonical as null

conversion status = failed

8.4 observation_latest

Use one table for both stations and reaches, but exactly one of station_id or reach_id must be populated.

observation_latest (
  entity_type text not null, -- station | reach
  station_id text references stations(station_id),
  reach_id text references reaches(reach_id),
  property text not null,
  observed_at timestamptz not null,
  value_native double precision,
  unit_native text,
  value_canonical double precision,
  unit_canonical text,
  quality_code text,
  quality_score double precision,
  aggregation text,
  is_provisional boolean not null default false,
  is_estimated boolean not null default false,
  is_missing boolean not null default false,
  is_forecast boolean not null default false,
  is_flagged boolean not null default false,
  provider_observation_id text,
  ingested_at timestamptz not null default now(),
  raw_payload jsonb not null,
  primary key (entity_type, coalesce(station_id, ''), coalesce(reach_id, ''), property)
)

In practice, PostgreSQL primary keys can’t use coalesce directly like that in a straightforward table definition, so implement with:

surrogate latest_id primary key, and

unique partial indexes:

unique on (station_id, property) where station_id is not null

unique on (reach_id, property) where reach_id is not null

That is the cleaner implementation.

11. Initial Provider Implementations
11.1 USGS adapter

Use USGS for observed U.S. stations and timeseries.

USGS Water Services supports:

site metadata

instantaneous values

daily values

JSON/XML/tabular outputs, and the newer USGS Water Data APIs expose OGC-compliant interfaces for monitoring locations and time series.

Initial implementation:

metadata from site service or OGC monitoring locations

latest discharge and stage from instantaneous values

historical backfill from daily values and, where needed, IV series

parameter-code mapping to canonical properties

Canonical mappings:

discharge: USGS streamflow parameter → discharge

gage height: → stage

Important:

preserve USGS qualification/provisional indicators

store datum metadata where available

do not assume all sites are rivers; filter to stream/river-relevant stations

11.2 Environment Agency adapter

Environment Agency real-time API exposes:

flood warnings

flood areas

water levels and flows

station metadata

measures and readings, and the docs explicitly recommend polling the single latest readings endpoint every 15 minutes rather than station-by-station crawling.

Hydrology API provides:

historic and recent hydrological data

near 8,000 monitoring stations

15-minute and daily flow/level series

quality-checked archives.

Initial implementation:

metadata from station endpoints

latest readings from real-time API

warnings from flood-monitoring warning endpoints

historical archives from hydrology API

Canonical mapping:

level → stage

flow → discharge

11.3 GEOGLOWS adapter

GEOGLOWS should populate modeled river reaches, not stations.

Use it for:

global modeled reach metadata

retrospective discharge

forecast discharge

return periods / threshold-like modeled context

GEOGLOWS provides forecast and retrospective streamflow services and advises using REST only for small request volumes, with AWS/S3 for larger-scale access.

Implementation rules:

source_type = modeled

entity stored in reaches, not stations

forecast rows must have is_forecast = true

never expose GEOGLOWS modeled flows in station endpoints

11.4 WHOS adapter

WHOS should initially be used for:

provider discovery

metadata enrichment

future pipeline seeding

Do not treat WHOS as a uniform observational feed. WHOS is a brokering/discovery framework linking national hydrological services and interoperable data services.

Implementation:

optional discovery crawler

registry enrichment

candidate-provider seeding table

12. Ingestion Workflows
12.1 Metadata sync

Frequency:

daily by default

weekly for slower providers if needed

Steps:

fetch provider catalog

normalize entity metadata

upsert stations or reaches

update last_metadata_refresh_at

mark previously known entities inactive if absent for repeated syncs

archive raw fetch payload

Rules:

never hard-delete entities

use status transitions:

active

inactive

discontinued

unknown

12.2 Latest observation sync

Frequency:

every 5–15 minutes depending on provider cadence

Steps:

fetch latest values

normalize timestamps

normalize property names

convert units

normalize quality flags

UPSERT into observation_latest

append to observation_timeseries

deduplicate same-timestamp replays

Deduping key:

entity + property + observed_at + provider_observation_id if available

12.3 Historical sync

Two modes:

backfill

rolling repair

Backfill:

when a new provider or entity is added

Rolling repair:

last 1–7 days to pick up delayed revisions

This matters because some providers revise historical readings after initial publication; EA explicitly notes that the latest-only polling pattern is not sufficient for a complete historical trace, and recommends using since style historical retrieval for completeness.

12.4 Threshold sync

Frequency:

daily or weekly

Steps:

fetch official thresholds where available

normalize severity taxonomy

convert units

store as versioned threshold rows

12.5 Warning sync

Frequency:

every 5–15 minutes

Steps:

fetch active warnings

normalize severity and status

store geometry if available

link to nearby or declared stations

close expired warnings when no longer active

13. Derived Thresholds

When official thresholds are absent, create derived thresholds from historical distributions.

Method:

require minimum historical coverage

compute percentiles on canonical values

exclude missing/flagged/forecast records

preferably compute by day-of-year seasonal window for strong seasonality

optionally compute all-time and seasonal variants

Initial derived types:

derived_p50

derived_p75

derived_p90

derived_p95

derived_p99

Recommended minimum data rule:

at least 1 year of usable data for crude percentiles

3+ years preferred

mark low-confidence derived thresholds explicitly

Store:

method

sample size

date range used

exclusion rules

14. Quality Normalization Rules

Use provider-specific mapping tables.

Example logic:

USGS provisional indicators → provisional

modeled forecasts → forecast

null reading or known no-data code → missing

suspect/flagged provider code → flagged

Also store:

provider raw code

normalized code

optional numeric quality score from 0 to 1

16. Spatial Query Design

If using PostGIS:

store points in geom

use GIST indexes

support bbox search with ST_Intersects

warning polygons can be stored in geometry column

reaches can later use LineString or representative point

For non-PostGIS fallback:

keep lat/lon columns

do bbox filtering numerically

PostGIS is worth using.

18. Data Validation Rules

Reject or quarantine records when:

lat/lon invalid

timestamp invalid

property unmapped

unit unmapped

observed value impossible by configured sanity rules

duplicate provider IDs with inconsistent metadata

Soft sanity checks:

negative discharge usually flagged, not always rejected

absurd stage spikes flagged

future timestamps flagged unless forecast

21. Security and Governance

Because this is a backend data platform:

secrets in env vars

no hardcoded tokens

provider auth abstraction

rate limiting on public API

usage logging

24. Non-Negotiable Implementation Rules

observed and modeled must remain separate in both storage and API responses

every normalized record must retain raw upstream payload

canonical units must always be stored when conversion is possible

latest sync must also append to historical store

provider adapters must be isolated modules

time series must be idempotent and deduplicated

warnings and thresholds must be versioned, not overwritten destructively

station and reach metadata must never be silently discarded

the API must expose provenance fields

all provider-specific logic must stay out of the API layer

26. Final Recommendation

The cleanest production architecture is:

observed gauges: USGS + Environment Agency first

modeled reaches: GEOGLOWS first

discovery layer: WHOS for future provider expansion

database: PostgreSQL + PostGIS

API: FastAPI

jobs: separate scheduled ingestion tasks

history: partitioned timeseries tables

normalization: central unit, quality, and vocabulary libraries

That design matches how the upstream systems are actually exposed today: EA splits near-real-time monitoring from historical hydrology archives, GEOGLOWS separates small-request REST access from bulk download workflows, USGS is in a migration period with both Water Services and newer OGC APIs, and WHOS functions as an interoperability/discovery framework rather than a single gauge feed.
